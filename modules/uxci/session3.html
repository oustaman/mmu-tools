Nielsen's 10 Usability Heuristics

The gold standard for UX evaluation - created by Jakob Nielsen in 1994

1. Visibility of system status
2. Match system & real world
3. User control and freedom
4. Consistency and standards
5. Error prevention
6. Recognition over recall
7. Flexibility and efficiency
8. Aesthetic and minimalist design
9. Help users recognize errors
10. Help and documentation

We'll explore each through real examples

Principle 1: Visibility of System Status

What it means: Always tell users what's happening

‚úÖ Good Examples:

  • ‚Ä¢ Loading spinner when uploading
  • ‚Ä¢ Progress bar showing % complete
  • ‚Ä¢ "Message sent" confirmation
  • ‚Ä¢ Battery indicator on phone

‚ùå Bad Examples:

  • ‚Ä¢ Button does nothing when clicked
  • ‚Ä¢ No feedback on form submission
  • ‚Ä¢ Mystery loading (how long?)

For your project: Show when buttons are pressed, actions are processing, or tasks complete

What it means: Use familiar language and concepts, not jargon

Principle 3: User Control and Freedom

What it means: Let users undo mistakes and change their minds

‚úÖ Good Examples:

  • ‚Ä¢ Undo/Redo buttons
  • ‚Ä¢ Cancel button on forms
  • ‚Ä¢ Back button that works
  • ‚Ä¢ Gmail's "Undo send" option

‚ùå Bad Examples:

  • ‚Ä¢ Can't go back without losing work
  • ‚Ä¢ No way to cancel ongoing action
  • ‚Ä¢ Permanent delete with no warning
  • ‚Ä¢ Forced linear flows

For your project: Always provide an escape route for users

Principle 4: Consistency and Standards

What it means: Same actions should work the same way throughout

‚úÖ Good Examples:

  • ‚Ä¢ Blue links throughout site
  • ‚Ä¢ Primary button always same color
  • ‚Ä¢ Icons mean same thing everywhere
  • ‚Ä¢ Same navigation structure

‚ùå Bad Examples:

  • ‚Ä¢ Submit button changes location
  • ‚Ä¢ X means close on one page, delete on another
  • ‚Ä¢ Different navigation per page
  • ‚Ä¢ Inconsistent terminology

For your project: Keep buttons, colors, and patterns consistent across screens

Principle 5: Error Prevention

What it means: Design to prevent errors before they happen

‚úÖ Good Examples:

  • ‚Ä¢ Disable submit until form valid
  • ‚Ä¢ Confirmation dialog for delete
  • ‚Ä¢ Auto-save drafts
  • ‚Ä¢ Constraints on date pickers

‚ùå Bad Examples:

  • ‚Ä¢ Can submit invalid forms
  • ‚Ä¢ No confirmation on destructive actions
  • ‚Ä¢ Easy to accidentally trigger
  • ‚Ä¢ No validation until after submit

For your project: Think about what could go wrong and prevent it

Principle 6: Recognition Over Recall

What it means: Show options rather than making users remember

‚úÖ Good Examples:

  • ‚Ä¢ Dropdown menus vs typing
  • ‚Ä¢ Icons with labels
  • ‚Ä¢ Recent search history
  • ‚Ä¢ Previously entered addresses

‚ùå Bad Examples:

  • ‚Ä¢ Unlabeled icons
  • ‚Ä¢ Hidden keyboard shortcuts only
  • ‚Ä¢ Must remember exact syntax
  • ‚Ä¢ No contextual help

For your project: Make options visible, don't hide them

Accessibility as a Principle

What it means: Design for everyone, including people with disabilities

‚úÖ Good Practices:

  • ‚Ä¢ High color contrast (text readable)
  • ‚Ä¢ Large enough tap targets (44x44px min)
  • ‚Ä¢ Alt text for images
  • ‚Ä¢ Keyboard navigation works
  • ‚Ä¢ Clear focus indicators

‚ùå Common Mistakes:

  • ‚Ä¢ Light gray text on white
  • ‚Ä¢ Tiny buttons hard to tap
  • ‚Ä¢ Color as only indicator
  • ‚Ä¢ Auto-playing videos with sound
  • ‚Ä¢ No keyboard access

For your project: Accessibility is often the easiest principle to demonstrate in sketches
