üéØ Five Whys Technique

Find the root cause, not just the symptom

üí° The Core Idea

Keep asking "Why?" to move from surface-level symptoms to the underlying root problem. Most initial problem statements describe what you see, not why it's happening.

üòµ Symptoms (What you see)

  • "Users don't click the button"
  • "People abandon the checkout"
  • "No one uses this feature"
  • "Complaints about the app"

üéØ Root Problems (Why it happens)

  • "Button isn't visible enough"
  • "Checkout has too many steps"
  • "Feature doesn't solve user needs"
  • "App is too complicated"

How to Use the Five Whys

1 Start with the symptom or surface problem you observe
2 Ask "Why does this happen?" and write the answer
3 Take that answer and ask "Why?" again
4 Repeat 3-5 times until you reach the root cause
5 The root cause is something actionable you can design for

üìù Example: Library Study Space Problem

Symptom: Students complain they can't find study spaces
‚Üì Why? Because they don't know which rooms are available
‚Üì Why? Because the booking system shows all rooms, not just available ones
‚Üì Why? Because students must check each room individually
‚Üì Why? Because there's no real-time availability filter
‚Üì Why? Because the system was designed to show all rooms by default, not to help find available ones quickly
üéØ Root Cause: The system prioritizes showing comprehensive information over helping users accomplish their task (finding an available space now)
